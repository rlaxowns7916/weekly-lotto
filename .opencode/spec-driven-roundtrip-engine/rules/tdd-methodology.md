# TDD 방법론

compile 시 코드 구현의 기본 작업 방식. Kent Beck의 TDD와 Tidy First 원칙을 따른다.

Red 테스트는 CLAUDE.md의 `행동 시나리오`, `오류 계약`, `수용 기준`에서 직접 도출한다.

## 핵심 사이클: Red -> Green -> Refactor

1. **Red**: 실패하는 테스트를 하나 작성한다.
   - 테스트 이름은 행동을 서술한다 (예: `shouldSumTwoPositiveNumbers`).
   - 실패 메시지가 명확해야 한다.
   - 시나리오 ID(`SCN-xxx`)와 테스트 이름을 1:1로 매핑한다.

2. **Green**: 테스트를 통과하는 최소한의 코드를 작성한다.
   - 최소 구현만 한다. 그 이상은 하지 않는다.
   - 모든 기존 테스트도 통과해야 한다.

3. **Refactor**: 구조를 개선한다.
   - 테스트가 통과하는 상태에서만 리팩터링한다.
   - 중복 제거, 명확성 향상에 집중한다.
   - 리팩터링 후 모든 테스트를 다시 실행한다.

한 번에 하나의 테스트만 추가하고, 모든 테스트 케이스가 완료될 때까지 사이클을 반복한다.
매 사이클마다 전체 테스트를 실행한다.

## Tidy First: 변경 분리

모든 변경을 두 유형으로 구분한다:

- **구조 변경**: 행동을 바꾸지 않는 코드 정리 (이름 변경, 메서드 추출, 코드 이동).
- **행동 변경**: 기능 추가 또는 수정.

원칙:
1. 구조 변경과 행동 변경을 섞지 않는다.
2. 둘 다 필요하면 구조 변경을 먼저 한다.
3. 구조 변경 전후로 테스트를 실행하여 행동이 바뀌지 않았음을 확인한다.
4. 보고서에 구조 변경 파일과 행동 변경 파일을 구분하여 기록한다.

## 결함 수정 시

1. 결함을 재현하는 실패 테스트를 먼저 작성한다.
2. 테스트를 통과하도록 수정한다.
