# SRTE Operational Rules

다음 규칙은 SRTE의 모든 작업(compile/decompile/verify/plan)에 공통 적용한다.

## 1) 필수 도구: serena-mcp

1. `serena-mcp`는 필수 도구다.
2. 작업 시작 전에 `serena-mcp` 사용 가능 여부를 먼저 확인한다.
3. `serena-mcp`가 없거나 비활성 상태면 즉시 중단하고 설치/활성화를 안내한다.

진단 경로 통일(serena 전용):
1. 구조/참조/영향 분석은 `serena-mcp` 경로만 사용한다.
2. `lsp_diagnostics`를 SRTE 검증 경로로 사용하지 않는다.
3. 품질 검증은 `serena-mcp` 분석 + 빌드/테스트/참조 일관성 점검으로 수행한다.

## 2) 도구 사용 우선순위

경계 탐색, 디렉토리 확인, 파일 점검은 `serena-mcp`를 우선 사용한다.

용도별 지침:
- **경계 탐색**: 디렉토리 구조 파악, `CLAUDE.md`/`IMPLEMENT.md` 존재 확인.
- **코드 구조 분석**: 심볼 검색, 참조 추적, 정의 이동으로 경계 내 구조와 경계 간 의존 관계 파악.
- **변경 영향 분석**: 참조 추적, 호출 계층 분석으로 변경 범위 파악.
- **경계 제외 규칙**: 기본 제외 목록과 사용자 정의 제외 목록은 `boundary-definition.md`를 따른다 (하드코딩 금지).

규칙 단일 정의 원칙(SSOT):
- 경계 정의/판정/제외 규칙: `boundary-definition.md`
- 문서 스키마/버전 규칙: `document-schema.md`
- 실행/대상 선택/보고 규칙: `operational-rules.md`

문서 우선 원칙(Source of Truth):
- CLAUDE.md는 "무엇을 보장하는가"를 고정하는 계약 문서다.
- IMPLEMENT.md는 "어떻게 만족시키는가"를 고정하는 설계/구현 문서다.
- compile은 문서 계약을 기준으로 Red 테스트를 먼저 작성해야 하며, 코드 구현은 계약 충족 수단으로 취급한다.
- CLAUDE.md는 헤더 파일(`.h`)처럼 외부 보장(계약값)을 고정하고, IMPLEMENT.md는 소스 파일(`.c`)처럼 내부 충족 경로를 추적 가능하게 기록한다.

PRD 품질 게이트(plan Phase 1.5):
- `PQG-1` 목적/요구사항 명확성: 모호어 없이 검증 가능한 문장으로 작성되어야 한다.
- `PQG-2` 경계 구조/의존성 결정성: 경계 목록, 경계 유형, 호출/참조 방향, 외부 연동 대상이 식별 가능해야 한다.
- `PQG-3` 시나리오 실행 가능성: Given/When/Then + 관측 가능한 단정값(필드/연산자/기대값)을 포함해야 한다.
- `PQG-4` 오류/수용 기준 테스트 환원성: 오류 코드/status/retry/조건과 체크리스트가 테스트로 환원 가능해야 한다.
- `PQG-5` 비기능 수치화: timeout, 성능, 동시성 등 비기능 요구가 정량/결정형 규칙이어야 한다.
- `PQG-6` 미확정 항목 관리: `오픈 질문` 항목에 `확인 불가 사유`/`확인 경로`/`해소 조건`이 있어야 한다.

Plan Phase 전환 불변식:
- `Phase 1 -> Phase 2` 전환 조건은 `사용자 승인 AND PQG-1~PQG-6 통과`다.
- PQG 실패가 하나라도 있으면 `Phase 1`에 머물며 PRD 보강 후 재판정한다.

상류 회귀(Upstream rollback) 규칙:
- verify/compile 단계에서 `QG-1~QG-4` 실패 원인이 PRD 모호성으로 판정되면 회귀 코드 `UPSTREAM-PRD-DRIFT`를 기록한다.
- `UPSTREAM-PRD-DRIFT` 발생 시 `plan Phase 1`로 복귀해 PRD를 우선 수정한다.

대상 확정 규칙(compile/decompile/verify 공통):
- 우선순위는 `<경로>` > `--all` > 인자 없음.
- 인자 없음: 커밋되지 않은 변경 파일별 경계를 계산하고 중복 제거한 경계를 기본 대상으로 한다.
- `<경로>` 지정: 지정 경계만 대상으로 하며 하위 경계를 자동 포함하지 않는다.

명령별 `--all` 규칙:
- decompile `--all`: 경계 판정 알고리즘으로 식별되는 전체 경계를 대상으로 한다.
- compile `--all`: `CLAUDE.md`가 있는 전체 경계를 대상으로 한다.
- verify `--all`: `CLAUDE.md`가 있는 전체 경계를 대상으로 한다.

명령별 인자 없음 보정 규칙:
- compile 인자 없음: 기본 대상(변경 파일 기반 경계)만 사용한다.
- verify 인자 없음: 기본 대상(변경 파일 기반 경계)만 사용한다.
- decompile 인자 없음: 기본 대상에 더해, 각 대상 경계의 상위 조상 중 `CLAUDE.md`가 있는 경계를 모두 포함한다(중복 제거).
- decompile 인자 없음의 목적: 상위 경계 문서의 `Schema-Version` 누락을 방지하고 스키마 마이그레이션 대상을 닫힌 집합으로 만든다.

문서 완전성 불변식:
- 모든 식별 경계는 `CLAUDE.md`와 `IMPLEMENT.md`를 모두 가져야 한다.
- 모든 경계 문서는 최신 `Schema-Version`(`.opencode/spec-driven-roundtrip-engine/rules/schema-versions/CURRENT.md` 값)을 기록해야 한다.
- 모든 `CLAUDE.md`는 스키마의 필수 계약 섹션을 모두 포함해야 한다.
- decompile은 대상 경계를 문서화 완전 상태로 수렴시켜야 한다.
- compile은 문서화 완전 상태가 아닌 경계를 대상으로 진행하지 않는다.

문서 품질 게이트(compile 시작 전):
- 각 행동 시나리오가 테스트 이름으로 1:1 매핑 가능해야 한다.
- 입력/출력/오류 계약이 타입/필수성/유효성 규칙 수준으로 모호함 없이 정의되어야 한다.
- 불변식/제약/비기능 요구가 수치 또는 명시적 규칙으로 정의되어야 한다.
- IMPLEMENT.md의 결정성 강화 섹션 포함 트리거가 활성화되면(아래 규칙 참조) 해당 섹션은 필수다.
- 하나라도 불충족이면 compile을 시작하지 않고 문서 보강을 우선한다.

문서 품질 게이트 세부 판정(검증 가능한 계약값):
- 시나리오-테스트 1:1 매핑 판정:
  - 각 `SCN-xxx`는 고유 ID를 가져야 하며 중복 ID를 허용하지 않는다.
  - 각 시나리오는 Given/When/Then을 모두 포함해야 한다.
  - Then은 관측 가능한 단정값을 포함해야 한다(필드, 연산자, 기대값).
    - 예: `status=APPROVED`, `errorCode=E_DUPLICATE_PAYMENT`, `attemptCount<=3`
  - 각 시나리오에 정확히 1개의 테스트 이름 후보를 매핑할 수 있어야 한다.
- 인터페이스/오류 계약 명확성 판정:
  - 입력 계약: 타입, 필수/옵션, 유효성 규칙(범위/포맷/열거형)이 모두 명시되어야 한다.
  - 출력 계약: 성공/실패 케이스별 필수 필드와 의미가 명시되어야 한다.
  - 오류 계약: `error code`, `HTTP status`(해당 시), `재시도 가능 여부`, `발생 조건`이 모두 있어야 한다.
- 제약/비기능 수치화 판정:
  - 최소 1개 이상의 정량 또는 결정형 규칙이 포함되어야 한다.
  - 허용 형태: 시간(`timeout<=1500ms`), 성능(`p95<=800ms`), 동시성(`same paymentId requests are serialized`), 멱등성(`same idempotencyKey returns same result`), 순서 보장 규칙.
  - 금지 형태: 검증 불가능한 서술형 문장(예: "충분히 빠르게", "안정적으로 처리").
- 결정성 강화 섹션 포함 트리거 판정:
  - 아래 중 하나라도 충족하면 트리거 활성화(`ON`)로 본다.
    - CLAUDE.md의 `SCN-xxx` 개수 >= 3
    - 대상 경계의 소스 구현 파일 수 >= 5 (테스트/문서/생성 산출물 제외)
    - 외부 의존/연동 대상 수 >= 2
  - 트리거 비활성화(`OFF`) 경계에서는 결정성 강화 섹션이 선택이다.
- 구현 추적성/결정성 판정(조건부):
  - 트리거 `ON`이면 `## 시나리오 추적성`/`## 파일 계약`/`## 변경 규칙` 섹션이 모두 존재해야 한다.
  - `## 시나리오 추적성`이 존재하면 CLAUDE.md의 모든 `SCN-xxx`를 누락 없이 포함해야 한다.
  - `## 시나리오 추적성`이 존재하면 각 `SCN-xxx`에 `구현 파일#심볼`과 `테스트명`이 1:1로 기록되어야 한다.
  - `## 파일 계약` 섹션이 존재하면 메서드 계약의 입력/출력/오류/제약 칼럼을 비우지 않는다.
  - `## 변경 규칙` 섹션이 존재하면 MUST/MUST NOT/함께 수정할 테스트 목록을 비우지 않는다.
  - 트리거 `OFF` 경계에서 위 섹션이 존재하지 않으면 `QG-5`를 적용하지 않는다.
- 확인 불가/오픈 질문 관리 판정(항상 적용):
  - `확인 불가` 또는 `오픈 질문` 항목이 존재하면 각 항목에 `확인 불가 사유`, `확인 경로`, `해소 조건`을 함께 기록한다.
  - `오픈 질문: 없음`으로 기록된 경우 해당 판정은 통과로 본다.

사실/해석 판정 규칙(decompile):
- `F1(사실-직접)`: 코드/설정의 리터럴, 명시적 상수, 정적 선언에서 직접 읽은 값
- `F2(사실-결정형)`: 코드 흐름 + 설정 키/기본값 조합으로 결정 가능한 값
- `F3(해석/의도)`: 코드에서 직접 증명되지 않는 설계 의도/정책 추론
- 문서 본문에는 `F1`, `F2`만 기록한다. `F3`는 반드시 `오픈 질문`으로 기록한다.

가정 안전성 판정(compile 완료 전):
- 가정은 `BLOCKING`/`NON-BLOCKING`으로 분류해 기록한다.
- `BLOCKING` 가정이 1개 이상이면 실패다.
- `NON-BLOCKING` 가정이 3개 이상이면 실패다.
- 가정이 없으면 `없음`으로 기록한다.

문서 품질 게이트 판정 코드:
- `QG-1` 시나리오 매핑 실패: `SCN-xxx` 고유성/Given-When-Then/Then 단정값/1:1 후보 매핑 조건 중 하나라도 불충족.
- `QG-2` 인터페이스 계약 실패: 입력 타입/필수성/유효성 또는 출력 필수 필드 정의 누락.
- `QG-3` 오류 계약 실패: 오류 코드, status, 재시도 가능 여부, 발생 조건 중 하나라도 누락.
- `QG-4` 제약/비기능 실패: 수치 또는 결정형 규칙 부재, 또는 금지 서술형 문장만 존재.
- `QG-5` 구현 추적성 실패(조건부): 결정성 강화 트리거가 `ON`인데 섹션이 누락되었거나, SCN/파일#심볼/테스트명 또는 메서드 계약 칼럼이 누락된 경우.
- `QG-6` 확인 불가 관리 실패: `확인 불가/오픈 질문` 항목이 존재하지만 사유/경로/해소 조건 중 하나라도 누락된 경우.
- `QG-7` 가정 안전성 실패(compile 전용): `BLOCKING` 가정이 존재하거나 `NON-BLOCKING` 가정이 3개 이상인 경우.
- compile/decompile/verify 보고서는 적용 가능한 실패 항목을 코드(`QG-1`~`QG-7`)로 기록한다.

문서 품질 게이트 실패 보고 규칙:
- compile은 실패한 경계를 즉시 중단하되, 다른 경계는 계속 진행한다.
- 실패 경계마다 다음을 반드시 보고한다.
  - 실패 항목: `시나리오 매핑`, `인터페이스/오류 계약`, `제약/비기능`, `구현 추적성/확인 불가 관리`, `가정 안전성` 중 어떤 게 실패했는지
  - 구체 사유: 어떤 `SCN-xxx`/계약 항목이 왜 검증 불가능한지
  - 최소 보강안: 문서에 추가해야 할 필드/규칙/수치 예시

변경 파일 기반 경계 계산(compile/decompile/verify 기본 모드에 적용):
- 파일 디렉토리에서 시작해 상위 조상으로 이동하며 첫 `CLAUDE.md` 디렉토리를 선택한다.
- 상위에 `CLAUDE.md`가 없고 현재 파일 디렉토리가 경계 판정 알고리즘을 만족하면 미문서화 경계로 처리한다.

decompile 인자 없음의 상위 경계 포함 규칙:
- 기본 대상 경계에서 시작해 상위 조상으로 이동하며 `CLAUDE.md`가 존재하는 경계를 모두 대상에 추가한다.
- 제외 디렉토리 하위 경계는 추가하지 않는다.
- 검증/보고 범위는 확장된 전체 대상 경계를 기준으로 한다.

## 3) 실행 제약

1. 최소 변경 원칙을 지키고, 요청과 무관한 리팩터링은 금지한다.
2. 추측으로 채우지 않고, 불명확한 항목은 오픈 질문으로 남긴다.
3. compile: `tdd-methodology.md`에 따라 테스트가 필수다. decompile: 가능하면 변경 범위에 대해 빌드/린트를 수행하고 결과를 보고한다.
