# Drift Check

경계 문서와 실제 코드 구현 간의 드리프트를 탐지한다.
문서 위치/경계 판정/제외 규칙은 `boundary-definition.md`를 따른다.

다음 규칙을 먼저 읽고 준수한다.

@.opencode/spec-driven-roundtrip-engine/rules/boundary-definition.md
@.opencode/spec-driven-roundtrip-engine/rules/operational-rules.md
@.opencode/spec-driven-roundtrip-engine/rules/drift-detection.md
@.opencode/spec-driven-roundtrip-engine/rules/document-schema.md

## 대상 선택

- 대상 선택 우선순위/기본 모드/예외 규칙은 `operational-rules.md`를 따른다.

## 점검 절차

1. 대상 경계를 식별한다.
2. 문서 완전성을 먼저 점검한다:
   - `CLAUDE.md` 또는 `IMPLEMENT.md`가 누락된 경계는 **높음 심각도** 드리프트로 기록한다.
   - `Schema-Version` 누락 또는 최신 버전(`.opencode/spec-driven-roundtrip-engine/rules/schema-versions/CURRENT.md` 기준) 불일치 경계는 **높음 심각도** 드리프트로 기록한다.
   - 필수 계약/구현 섹션 누락 경계는 **높음 심각도** 드리프트로 기록한다.
3. 각 경계의 CLAUDE.md를 읽고 코드와 대조한다:
    - 기능 범위/비범위가 코드와 일치하는가?
    - 공개 인터페이스 계약(입력/출력/유효성)이 실제와 일치하는가?
    - 행동 시나리오/오류 계약/수용 기준이 코드와 테스트에서 검증 가능한가?
    - 불변식/비기능 요구를 코드가 준수하는가?
4. 각 경계의 IMPLEMENT.md를 읽고 코드와 대조한다:
    - 모듈 분해/호출 흐름/핵심 알고리즘이 실제와 일치하는가?
    - 데이터 모델/외부 연동 정책/설정이 실제와 일치하는가?
    - 예외 처리/관측성/테스트 설계가 실제 구현과 일치하는가?
5. 코드/문서에서 판정이 불가능하거나 추가 확인이 필요한 항목은 `오픈 질문`으로 정리한다.
   - 각 항목에 `확인 불가 사유`, `확인 경로`, `해소 조건`을 함께 기록한다.
6. 드리프트를 심각도별로 분류하여 보고한다.

## 출력 형식

```
# 드리프트 체크 보고서

## {경계 이름}
| 유형 | 심각도 | 내용 |
|------|--------|------|
| 책임 불일치 | 높음 | {설명} |
| 구조 불일치 | 중간 | {설명} |

- 오픈 질문: {없음|추가 확인이 필요한 사항}
  - 항목 형식:
    - 내용: {질문/확인 불가 항목}
    - 확인 불가 사유: {현재 판정 불가 이유}
    - 확인 경로: {확인할 파일/로그/테스트/담당자}
    - 해소 조건: {종결 기준}

## 요약
- 점검 경계: {N}개
- 드리프트 발견: {T}건 (높음 {H}, 중간 {M}, 낮음 {L})
- 드리프트 없음: {K}개 경계
- 오픈 질문: {Q}건
```

## 제약

1. 파일을 생성/수정/삭제하지 않는다.
2. 드리프트 수정은 하지 않는다. 보고만 한다.
3. 코드에서 확인할 수 없는 사항은 추측하지 않고 "확인 불가"로 표시한다.
4. `확인 불가`로 표시한 항목은 반드시 사유/경로/해소 조건을 함께 남긴다.
